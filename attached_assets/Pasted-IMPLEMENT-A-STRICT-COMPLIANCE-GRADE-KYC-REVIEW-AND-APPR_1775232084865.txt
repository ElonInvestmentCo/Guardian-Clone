IMPLEMENT A STRICT, COMPLIANCE-GRADE KYC REVIEW AND APPROVAL WORKFLOW THAT ENFORCES CONTROLLED USER STATE TRANSITIONS AND PREVENTS PREMATURE ACCESS TO THE PLATFORM.

When an administrator determines that a user must resubmit information, the system must update the user’s KYC record by setting the status to "resubmit_required", storing the specific fields that require correction, and attaching a clear, user-facing rejection reason. From that point forward, any user with a KYC status of "resubmit_required" must be automatically redirected to the dedicated /kyc/resubmit route upon login or page access, and must be prevented from accessing the dashboard or any protected platform features.

The /kyc/resubmit interface must dynamically render only the fields explicitly requested by the administrator, clearly display the rejection reason at the top of the page, and guide the user to correct only the necessary information. Previously submitted data should be pre-filled where applicable, and all validation must be restricted to the required fields. Upon submission, the system must securely overwrite the previously rejected data and immediately transition the user’s KYC status to "reviewing", while clearing any resubmission requirements.

Following a successful submission, the user must be automatically redirected to a dedicated /kyc/reviewing page. Under no circumstances should the user be redirected to the dashboard at this stage, nor should they be prompted to resubmit again. The /kyc/reviewing page must present a locked, read-only state indicating that verification is in progress, with messaging such as “Your information is currently being reviewed by our compliance team,” accompanied by a visible status indicator of “KYC Status: Reviewing.” All editing capabilities must be disabled, and access to trading, funding, or other restricted features must remain blocked.

The system must enforce strict route-level access control such that any user with a KYC status of "reviewing" is continuously redirected to /kyc/reviewing and cannot bypass this state. Only users with a KYC status of "approved" are permitted to access the /dashboard and full platform functionality.

When an administrator approves a user’s KYC, the system must update the status to "approved" and trigger a real-time state transition on the user side. This must be implemented using either WebSocket subscriptions or a polling mechanism at regular intervals. Once the system detects that the user’s KYC status has changed to "approved", the user must be automatically redirected from /kyc/reviewing to /dashboard, and presented with a confirmation message such as “Your account has been successfully verified.”

At all times, the system must maintain strict state integrity. Users must never be returned to the resubmission flow after successfully submitting corrections, must never gain access to the dashboard prior to approval, and must never encounter duplicate prompts or ambiguous states. The KYC lifecycle must remain deterministic and follow a clear progression: "resubmit_required" to "reviewing" to "approved".

The final implementation must reflect a seamless, enterprise-level user experience in which the user is always aware of their current status, required actions, and next steps, with zero confusion, zero redundancy, and full compliance with financial platform standards.
